# Audit — 1.5.x hardening pass, 2026-06-13

**Scope**: the **1.5.x surface** — everything the data-ingestion & curation arc added
that touches memory, the checkpoint format, or untrusted input: the **packed `g_data`
token store** (1.5.3 — the `g_data_w` width global, the `_tw_ld`/`_tw_st` /
`gd_ld`/`gd_st` accessors, `corpus_set`'s `alloc(len*g_data_w)`, the `bpe_learn`
u8→u16 self-widen, the `tok_encode` output-width arg) and the **raised
`MAX_CORPUS_BYTES` (4 MB → 64 MB)**; the **`c4_sample.py` curation script**
(1.5.1/1.5.2/1.5.4 — it ingests untrusted gzipped JSON-lines shards from a public
URL); the **char-diffusion path** (1.5.0 — `gen_diffusion`/`eval_diffusion`, masked-CE,
the `mask_emb`/bidirectional core, checkpoint v7), which had not been separately
audited; and **checkpoint back-compat + the additive/frozen-surface invariant +
memory** across the arc. This is the P(-1) hardening cut that **closes the 1.5.x arc**
before 1.6.0 / M16. **Method**: an adversarial multi-agent review — five independent
read-only dimensions, each medium-or-higher finding then adversarially verified
(refute-by-default) against the committed code — plus a captured benchmark baseline
and manual regression of the one fix. The **ninth** audit; builds on the
M2/M3/M5/M6/M7/M8, v1.0-final, and 1.4.x-hardening audits.

## Findings → dispositions (1 confirmed; fixed in this cut)

**Fixed in 1.5.5 — the one confirmed finding:**
- **Gzip-bomb: unbounded line buffering in `c4_sample.py`** (medium; `scripts/c4_sample.py`,
  the shard stream loop). The script ingests **untrusted** gzipped JSON-lines shards
  (the public `allenai/c4` HF URL, or arbitrary input via `--url -`) and iterated them
  with a plain `for line in gz:`. Python's `GzipFile` iteration calls `readline()` with
  no size bound, so it **materializes a whole line into one `bytes` object** before
  yielding — and DEFLATE compresses a single-byte run at ~1000:1, so a ~KB-scale
  compressed member with no `\n` decompresses into hundreds of MB in memory. The only
  loop bounds (`budget` / `--max-bytes`) are checked *after* the line is already
  buffered, so they cannot cap a single line → **OOM** on a malformed or MITM'd shard.
  (Adversarially verified real; downgraded high→medium: an offline operator-run
  data-prep CLI, trusted default source, single short-lived process, no corruption /
  persistence / escalation.) **Fix**: a bounded `iter_lines(gz, oversized)` that reads
  fixed 1 MB chunks and splits on `\n` locally, **dropping** any runaway no-newline
  line over an 8 MB cap (real C4 docs are a few KB) and resyncing at the next newline —
  memory is capped at ~chunk+max_line, the count is reported (`oversized=N`), and a
  valid doc *after* a bomb is still recovered. **Regression-verified** (no Cyrius
  change, so the binary + its 977 grad-checks are unaffected):
  1. a normal small gzip (3 docs) extracts all three, `oversized=0`;
  2. a 12 MB no-newline bomb + a trailing valid doc → exit 0 (no OOM), `oversized=1`,
     the trailing doc recovered;
  3. re-curating the 4 MB sample is **byte-identical** to the pre-fix corpus
     (sha `e0e693e2…`), so well-formed shards yield identical text — every keep/dedup
     decision, and thus X016/X017/X019 reproducibility, is preserved.

**Refuted (verified not-a-bug):**
- *Budget rebalancing can write past `--max-bytes`.* The one-document overshoot is
  real but is the documented **streaming-stop** semantic (collect-to-~target then
  stop), causes no overflow/OOM/OOB (Python big-ints, sequential text write), and
  cannot propagate downstream — the binary hard-bounds ingestion (`secure_read_file`
  reads ≤ cap; `read_stdin` rejects `>cap` with −2). Not a defect; no change.

**Advisory (low; accepted, no code change):**
- *Dedup sets grow with kept-doc count.* Bounded by `--max-bytes` (≤ N/min_doc_len
  hashes × 16 B ≈ ~5 MB at the 64 MB ceiling) — acceptable curated-sample overhead.
- *`--out` has no path-traversal / symlink guard.* Expected behavior for an
  operator-invoked, non-privileged dev CLI with an explicit `--out` flag; the file is
  written where the user asks.

## Cleanliness + benchmark baseline (P(-1))

- **Cleanliness**: `cyrius build` OK, `cyrius lint` 0 warnings, `cyrius test` **977
  passed / 0 failed** (x86_64 + aarch64/qemu) before and after the cut.
- **Benchmark baseline** captured (`./build/bench`): default fwd+bwd step
  **3,580,908 ns** (~3.58 ms), consistent with the flat 0.4.0→1.4.6 history. The one
  fix is **Python-side only** — the Cyrius binary is byte-identical across this cut, so
  the baseline is unaffected and there is no perf regression to prove.

## Coverage note

Four of the five dimensions returned **zero** findings with explicit per-item coverage
of the obvious classes (integer overflow, OOB, alloc-fail, width-decision, div-by-zero,
hostile checkpoint, format/back-compat, memory). The 1.5.3 packed store — the arc's
main new code — is confirmed sound. The single real issue was in the **dev-side
curation script's handling of untrusted external data**, now fixed. **Verdict: GO**,
the 1.5.x arc is closed.
