# Audit — 0.7.1 (M7: scale preset + BPE), 2026-06-11

**Scope**: the full 0.7.1 diff — E3 (opt-in simple BPE tokenizer `--bpe K`:
`bpe_learn`/`tok_encode`/`bpe_build_spans`/`tok_emit`; the `--preset` and
`--heads`/`--kv-heads`/`--layers` config surface; checkpoint format v3 with
the tokenizer triple + merge table and v1/v2 compat; `--eval` bits-per-byte)
plus the new tests/bench. Method: adversarial multi-agent review (6
dimensions — BPE learner, v3 loader hostile-input, encode/round-trip,
config/CLI, eval/bits-per-byte, cross-cutting memory/overflow — each raw
finding independently judged by 3 verifier lenses (correctness, exploit/repro,
scope) instructed to refute; ≥2/3 upholds). 9 agents; 1 raw finding → 1
confirmed, 0 refuted.

## Confirmed findings → fixes

1. **[high] `--layers` heap-OOB on the fresh-model path** (confirmed 3/3, one
   verifier with a live SIGSEGV repro). M7 made the model config
   CLI-controllable (`--heads`/`--kv-heads`/`--layers`), but `model_config_ok`
   bounded `NL` only by `≥ 1` — no upper bound — and, unlike the checkpoint
   loader, `model_init` did **not** run the `model_alloc_bytes` pre-allocation
   cap. A crafted `--layers` value (e.g. `8911182964448639662`, parsed exactly
   by `_atoi` — no `_atoi` overflow even required) made
   `g_NP = _emb_end() + NL·_blk() + 2C` overflow i64, wrapping the request to a
   small positive size; `alloc()` (reject only `≤ 0` or `> 256 MB`) accepted an
   **undersized** buffer, after which `t_alloc`'s zero-fill and the weight-init
   loops wrote far past it — heap corruption / SIGSEGV, reachable on the
   **default corpus with no `--load`**. Heads/kv-heads could not reach this
   (`model_config_ok` already caps them to `[1, C]` with the divisibility
   checks); only `NL` escaped bounding. This regressed the exact invariant the
   M6 audit established for the *file* path (loader `-8` NL ≤ 128 and `-18`
   `model_alloc_bytes` pre-flight) — the *CLI* path was the new, unguarded
   twin. **Fixed**: `model_config_ok` now caps `V/C/T/NL` to the same
   magnitudes the loader enforces (`-4/-5/-7/-8`: `V ≤ 768`, `C ≤ 4096`,
   `T ≤ 8192`, `NL ≤ 128`) — the `NL` cap keeps every later product below the
   i64 overflow point — and `model_init` runs `model_alloc_bytes(...) >
   CKPT_MAX_MODEL_BYTES()` (128 MB) before any allocation, a clean abort.
   The two config gates (checkpoint header and CLI) now share one invariant.
   Regression: `test_config_caps` (boundary `NL = 128` accepted / `129`
   rejected, huge `NL` rejected, and the `V/C/T` caps); the live repro now
   exits 1 with `model_init: invalid config` / `…exceeds the 128 MB
   model-allocation cap` instead of dumping core.

## Verification records (examined, found sound — not bugs)

The other five dimensions returned no confirmed defect. Recorded so the
surfaces are not re-litigated:

- **BPE learner expansion cap** — `bpe_learn`'s `TOKLEN` eligibility test
  (`la + lb ≤ 64`) sits *inside* the `(c > best)` argmax body, so an oversized
  pair can neither be selected nor raise `best` to shadow an eligible pair.
  No learned merge can mint a token whose expansion exceeds the
  `BPE_MAX_TOKLEN` span stride, so `bpe_build_spans` cannot write OOB. The
  pair-count index `(left·VMAX + right)` stays `< VMAX²` because `Vcur ≤ B + K
  ≤ 768 = VMAX`. The argmax scan (row-major ascending, strict `>`) is the
  frozen deterministic tie-break; the in-place greedy replace holds `wr ≤ rd`.
- **v3 loader hostile-input** — the tokenizer triple (`-32/-33/-34/-35/-36`),
  the well-founded-DAG merge check (`-37`: every ref in `[0, Vb+j)`, rejecting
  negative/self/forward refs and therefore all cycles), and the length
  recurrence (`-38`: every expansion ≤ 64 bytes) are all validated on fixed
  stack scratch **before** any allocation; integer products stay bounded by
  the magnitude caps checked first. v1/v2 images load as byte-level (kind 0,
  `Vb = V`, `K = 0`) and cannot reach `-32…-38`.
- **encode / round-trip determinism** — `tok_encode` replays the learned
  merges in mint order with the same greedy left-to-right non-overlapping
  semantics as `bpe_learn`, so re-encoding the original corpus reproduces
  `g_data` bit-for-bit (the resume corpus-rebuild relies on this;
  `test_bpe_roundtrip` pins it). Prompt encode is bounded by `ENC_MAX`.
- **eval / bits-per-byte** — `--eval` is RNG-neutral and dropout-off:
  independently confirmed that a `--save` is **bit-identical** with or without
  `--eval` (same SHA-256). For byte-level, bits/byte `= ce_nats / ln 2`; for
  BPE, targets are weighted by their byte expansion so the metric is
  tokenizer-comparable. `test_eval_bpb` pins it.
- **vocab > 256 propagation** — the `V ≤ 768` buffers (embeddings, tied head,
  logits/probs) are runtime-sized; the latent `ckpt_serialize` vocab-loop OOB
  (looped to `g_V` over the 256-entry `g_vocab`) was already fixed to run to
  the base-vocab count; `test_alloc_accounting` pins the bound at `V = 300`.

## Result

242 checks green on x86_64 and aarch64 (qemu); fuzz green (500 byte-level +
500 BPE mutated checkpoints + 100 random corpora + BPE round-trip); DCE
release build static-ELF green; agnos build green. The single confirmed
finding is fixed and regression-tested in this release.
